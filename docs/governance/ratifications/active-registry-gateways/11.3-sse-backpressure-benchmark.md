# Ratification 11.3 — Two-Hop SSE Backpressure Benchmark

**Gate for:** Release N+2 · **Spec:** [active-registry-gateways/tasks.md §11.3](../../../../openspec/changes/active-registry-gateways/tasks.md)
**Status:** Open · **Owner:** Platform Engineering
**Decision required by:** Release N+2 freeze date

## The question

> Benchmark two-hop SSE backpressure (`skill-gateway → a2a-gateway → external`) under load; capture the SLO and reject the release if p95 added latency > 50 ms.

## Why it matters

External developers using the public `skill-gateway` reach internal A2A agents through a two-hop SSE relay. Each hop has a bounded channel (default 32 events) to propagate backpressure end-to-end. If the per-event added latency exceeds 50 ms at p95 under expected load, the user experience degrades: incremental tool output starts to feel laggy and the deprecation telemetry shows symptoms operators would rather avoid.

## Benchmark protocol

Run from `services/a2a-gateway/tests/` (per-event harness, not yet checked in — produce in the N+1 → N+2 window):

1. Spin up:
   - One synthetic external agent emitting one SSE event every 5 ms for 60 s.
   - `a2a-gateway` with `SSE_BUFFER_SIZE=32`.
   - `skill-gateway` with the corresponding upstream pointed at `a2a-gateway`.
   - A single client subscribed through `skill-gateway`'s public endpoint.
2. Run a steady ramp from 1 → 100 concurrent streams over 5 minutes.
3. Capture:
   - End-to-end per-event latency (upstream emit ts → client receive ts).
   - p50, p95, p99 of the latency histogram.
   - Per-hop bytes-in / bytes-out.
   - Backpressure event count (the channel-full counter we add for this benchmark).

## Acceptance threshold

> The release is accepted iff **p95 added latency ≤ 50 ms** between the upstream emit timestamp and the client receive timestamp, at 100 concurrent streams.

If we exceed the threshold:

- Investigate the bounded-channel size; bumping `SSE_BUFFER_SIZE` increases throughput at the cost of higher memory per stream.
- Inspect `skill-gateway`'s connection pool — under load it may be serializing more than expected.
- Worst case: defer Release N+2 enforcement until the relay path is fixed; the compat shim continues to forward.

## What "added" means

The benchmark measures the latency the platform adds vs. a direct caller-to-upstream connection. A baseline pass (caller hits the synthetic agent directly) establishes the floor; the two-hop measurement is delta over that floor.

## Open items

- The benchmark harness itself is not yet checked in — produce in the N+1 window.
- Decide the load shape: is "100 concurrent streams" the right p95 target, or should we target a higher percentile?
