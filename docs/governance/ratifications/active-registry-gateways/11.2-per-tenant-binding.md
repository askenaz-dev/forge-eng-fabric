# Ratification 11.2 — Per-Tenant Single-Binding Artifact Store

**Gate for:** Release N+1 · **Spec:** [active-registry-gateways/tasks.md §11.2](../../../../openspec/changes/active-registry-gateways/tasks.md)
**Status:** Open · **Owner:** Platform Engineering
**Decision required by:** Release N+1 freeze date

## The question

> Can a Tenant configure two backends (e.g. Nexus for skills + GitHub Packages for prompt templates) or one binding per Tenant?

The design ([decision 3](../../../../openspec/changes/active-registry-gateways/design.md)) leans single-binding per Tenant for simplicity. Open until customer feedback during Release N+1 confirms.

## Proposed decision

Ship Release N+1 with **single-binding per Tenant**. The `artifact_store_binding` table has `tenant_id` as the primary key, enforcing this at the schema level. Per-asset-family bindings are deferred to a follow-up if and only if at least one pilot Tenant demonstrates a concrete need.

## Why single-binding (for now)

| Pro | Con |
|---|---|
| One credential rotation surface per Tenant | If a Tenant uses two stores in their existing stack, they have to consolidate before adopting Forge |
| Single audit trail per Tenant | Operators with strong opinions about per-tooling separation will push back |
| Simpler Portal UX | Some asset families don't fit naturally on certain backends (e.g. workflow-as-skill in CodeArtifact's `generic` format works but is awkward) |
| Schema enforces it; no race conditions | — |

## How to evolve if needed

Migration path from single → per-family is additive:

1. Add `artifact_store_binding_family(tenant_id, family text, backend text, config_json jsonb)` table.
2. Adapter factory consults the per-family table first; falls back to the per-Tenant single-binding table when no per-family row exists.
3. Portal **External integrations** view grows a per-family dropdown.

No data migration required for existing single-binding Tenants — they continue to fall through to the per-Tenant table.

## Customer-feedback gate

Release N+1 ships single-binding. We collect feedback over the first 90 days. If ≥ 2 Tenants ask for per-family bindings with a real use case (not theoretical), we open a follow-up spec.
