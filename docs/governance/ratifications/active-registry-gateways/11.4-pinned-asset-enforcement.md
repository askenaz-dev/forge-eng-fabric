# Ratification 11.4 — Pinned-Asset Enforcement: Hard-Fail vs Warn

**Gate for:** Release N+3 · **Spec:** [active-registry-gateways/tasks.md §11.4](../../../../openspec/changes/active-registry-gateways/tasks.md)
**Status:** Open · **Owner:** Platform Engineering + SDLC Team
**Decision required by:** Release N+3 freeze date

## The question

> When the LLM tries to invoke an unpinned asset on a pinned-set workflow, do we hard-fail or warn?

Default proposal: **hard-fail** with `403 asset_not_pinned`. The current implementation matches the proposal — the runtime refuses the invocation and emits `guardrail.trip.v1` with `reason=asset_not_in_pinned_set`.

## Why this is a question

A hard-fail is the strictest posture: the workflow stops. A warn-only posture would let the call through and emit a telemetry event, leaving cleanup to a human reviewing the audit trail. Both have failure modes:

| Posture | Strength | Weakness |
|---|---|---|
| Hard-fail | Pinning becomes a real contract; auditors can rely on it | Brittle when the LLM legitimately needs an off-pin asset; users may resent it as "the runtime won't let me do my job" |
| Warn-only | Soft landing; users keep moving | Pinning becomes documentation, not enforcement; audit gets noisy with `should_have_been_pinned` events |

## Recommendation

**Ship hard-fail** as the default at Release N+3. Pilot evidence from N+2 should inform whether to add an escape hatch.

### Escape hatch design (only if needed)

If pilot feedback demands an escape:

1. Add a `selected_assets.bypass_token` field to the workflow AST. Setting it to a signed token from a Workspace owner permits one-off off-pin invocations during that execution.
2. The runtime accepts the bypass token, emits `guardrail.trip.v1` with `reason=asset_off_pin_acknowledged`, and records the token id in the audit trail.
3. Tokens have a short TTL (default 1 hour) and are scoped to a single workflow id.

This keeps the hard-fail posture for normal traffic while giving operators a clear, audited path for exceptional cases.

## Pilot data to collect during N+2

- Count of `guardrail.trip.v1{reason=asset_not_in_pinned_set}` events per Tenant per day.
- Number of distinct workflows that hit the trip.
- Workspace-owner feedback on whether the hard-fail was a real problem or a teaching moment.

## Don't ship the warn-only posture by default

Warning-only undermines the spec's entire reason to add pinning. The whole point ([design.md decision 6](../../../../openspec/changes/active-registry-gateways/design.md)) is that pinning makes the LLM's tool-choice space bounded; if the bound is advisory, it's not a bound.
