Chapter 01 — What Forge is

A platform, not a sidecar

One fabric
woven from agents,
specs, policies and runs.

Modern engineering is fragmented. Specs live in a wiki, code lives in twelve repositories, policies live in a YAML file no-one reads, and incidents live in a Slack channel that fills with screenshots. Forge collapses that fragmentation into a single, auditable control plane — driven by a control-plane agent (Alfred), a structured specification format (OpenSpec), composable Workflows, and a Marketplace that ships them across teams.

Every action is policy-checked. Every decision is signed. Every workflow can be rolled back. The result is a system where humans set intent, agents execute, and governance is structural — not a quarterly audit.

  • Spec-driven. OpenSpec is the source of truth — code, deploys, and tests trace back to it.
  • Agentic by default. Alfred and specialised agents act within delegated permissions.
  • Policy-gated. OPA + OpenFGA enforce who can do what, where, when.
  • Human-in-the-loop, by configuration. Autonomy levels per OpenSpec, per Workspace.
  • Self-healing. Incidents → diagnosis → action → postmortem → evolution proposal.

Chapter 02 — The five pillars

The platform, distilled.

Five capabilities, woven together. Each is independently useful. Together they are the engineering fabric.

01 / Control plane

Alfred — the agent that listens.

A FastAPI agent that interprets natural-language intent, retrieves context via RAG on Milvus, evaluates OPA policy and drives every downstream tool exclusively through LiteLLM. Every decision is logged with the OpenSpec it traces back to.

FastAPI LiteLLM Milvus RAG OPA
02 / Source of truth

OpenSpec — every change is a spec.

Structured proposals with deltas, capabilities, scenarios. Versioned, signable, diff-able. The single contract between humans and agents. Specs link to code, traces, runbooks and FinOps attribution.

Spec deltas Versioned Linked artefacts
03 / Composition

Workflows & Skills.

Compose Skills, MCP tools, prompts, gates, branches and human-in-the-loop steps into versioned workflows. Publish to the Marketplace at workspace, tenant or forge-certified visibility.

Temporal MCP SemVer Marketplace
04 / Governance

Policy, approvals, audit.

OPA evaluates every action. OpenFGA scopes who can read or write what. Approval queues hold high-criticality work for the right humans. Every audit event is hash-chained, signed, and queryable.

OPA OpenFGA Hash chain HITL
05 / Autonomous ops

The self-healing loop.

Incidents detected → diagnosed → healed (within delegated permissions) → postmortem auto-generated → improvement proposals routed to the Evolution Inbox for human review. The platform learns from its own incidents.

Detection Diagnosis Healing Evolution
06 / Cost & portal

The tablero.

A bilingual ES/EN Portal with ⌘K navigation, KPI sparklines, live runs, signed audit timeline and a service-mesh visualisation — built on a token-driven design system. Everything wired to live platform data, never mocks.

Chapter 03 — The fabric

Built for engineering at scale

A hub-spoke mesh, by design.

The Forge orchestrator routes every intent across a small, opinionated set of services: a policy engine, an authorisation graph, an asset registry, a hash-chained audit log, a context engine that fuses RAG + traces, and a spec engine that holds every change.

Services talk over Temporal for durable workflows, Kafka for events, and gRPC for synchronous calls. Postgres holds canonical state. Milvus and pgvector hold RAG embeddings. Tempo + Loki + Prometheus stitch traces, logs and metrics into one observability surface.

The shape is deliberately boring — boring shapes are the ones that survive a decade in production.

Chapter 04 — From intent to deploy

Five steps. One trace.

From a sentence on a Tuesday morning to a signed, policy-gated deploy on prod by Wednesday. Every step is a span in one OpenTelemetry trace — auditable from prompt to pod.

01

Capture

A non-technical user describes the intent. Alfred asks follow-up questions and assembles a complete OpenSpec.

02

Scaffold

The scaffolder picks a template, generates the repo, opens a PR, and registers the new asset.

03

Gate

CI gates run — tests, SBOM, image-verify. OPA checks the change against the workspace policy.

04

Approve

If the change crosses a criticality threshold, an approver gets a typed approval request with diff, runbook and rollback plan.

05

Deploy

The deploy-orchestrator drives a staged rollout with canary, verify, notify. SLOs and FinOps attribution are wired automatically.

Chapter 05 — Trust, by design

Autonomy with brakes.

Autonomy is a slider, not a switch. Every workspace and every OpenSpec carries its own autonomy ceiling. The platform's defaults err on the side of human approval for irreversible operations.

Do

Let Alfred draft, refactor, propose, observe and self-heal — all within delegated permissions and recorded in audit.

Don't

Skip approvals for production data migrations, billing changes, IAM grants, or anything that can't be cleanly rolled back.
